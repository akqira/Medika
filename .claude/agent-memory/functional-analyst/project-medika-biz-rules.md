**BR-SCHED-01:** A new appointment MUST NOT overlap an existing appointment for the same doctor. The backend enforces this via IAppointmentRepository.HasConflictAsync.

**BR-MED-01:** A Consultation can only be finalized once (IsFinalized flag). Finalizing automatically: (1) marks the linked Appointment as Completed, (2) creates a Pending Invoice, (3) calls Patient.RecordVisit().

**BR-FIN-01:** An Invoice can only be marked Paid when its status is Pending. Paid invoices cannot be cancelled.

**BR-FIN-02:** Invoice number format is "F-YYYY-NNN" (e.g. F-2026-001), generated by IInvoiceRepository.GenerateNumberAsync.

**BR-PAT-01:** Allergies are case-insensitive deduplicated at the domain level. They must be displayed as warnings (⚠) on any prescription or appointment detail.

**BR-AUTH-01:** Only a Doctor-role user can register new users. The /api/auth/register endpoint is role-gated to Doctor.

**BR-AUTH-02:** Only a Doctor-role user can access consultation and financial summary endpoints. Receptionists are limited to patient and appointment management + invoice payment.

**BR-TARIFF-01:** Tariff (consultation fee) must be >= 0 DA. It is set per consultation, not globally configured (no default tariff feature yet).

**BR-APPT-01:** Appointment state machine: Pending -> Confirmed -> InProgress -> Completed. Cancel is allowed from any non-terminal state. NoShow is allowed from Pending or Confirmed. Reschedule resets to Pending.

**BR-DATA-01:** All timestamps stored in UTC. Frontend formats dates in fr-FR locale for display.

**Why:** These rules emerged from reading domain logic. They must be reflected in every acceptance criterion that touches the relevant aggregate.

**How to apply:** Cite the relevant BR code in Acceptance Criteria and Business Rules sections of user stories.
